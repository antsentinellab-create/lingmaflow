# Brainstorming - 需求釐清與設計討論

## 核心原則

**開始任何實作前先釐清需求**。這是整個開發流程中最重要的步驟，決定了後續工作的方向和品質。

### 為什麼需要 Brainstorming？

- ❌ **不要**假設你理解用戶的需求
- ❌ **不要**急著跳進實作細節
- ✅ **要**透過提問確保雙方理解一致
- ✅ **要**探索多種可能的解決方案

## 提問技巧

當用戶提出需求時，使用以下問題引導他們說清楚目標：

### 1. 目標與價值

```
- 這個功能要解決什麼問題？
- 誰會使用這個功能？（目標用戶）
- 成功的標準是什麼？
- 為什麼現在需要做這個？
```

### 2. 使用情境

```
- 描述一個典型的使用場景
- 用戶在什麼情況下會使用這個功能？
- 用戶的輸入是什麼？預期的輸出是什麼？
- 有哪些邊界情況需要考慮？
```

### 3. 功能範圍

```
- 必須有的功能（Must have）是什麼？
- 最好有的功能（Nice to have）是什麼？
- 哪些功能明確不在範圍內？
- 時間和資源限制是什麼？
```

### 4. 技術約束

```
- 有現有的系統或 API 需要整合嗎？
- 有效能、安全性或合規性要求嗎？
- 有偏好的技術棧或架構嗎？
- 需要向後相容嗎？
```

## 設計選項分析

當需求釐清後，列出可能的設計方案：

### 方案比較模板

```markdown
## 選項 A：[方案名稱]

**優點：**
- 優點 1
- 優點 2

**缺點：**
- 缺點 1
- 缺點 2

**適用情境：** 什麼時候選擇這個方案

## 選項 B：[方案名稱]

**優點：**
- 優點 1
- 優點 2

**缺點：**
- 缺點 1
- 缺點 2

**適用情境：** 什麼時候選擇這個方案

## 建議方案

推薦 [選項 X]，因為...
```

### 常見取捨維度

- **複雜度 vs 功能完整性**
- **開發速度 vs 可維護性**
- **效能 vs 記憶體使用**
- **靈活性 vs 簡單性**
- **短期利益 vs 長期技術債**

## 工作流程檢查清單

在進入實作之前，確認以下項目：

- [ ] 用戶痛點和目標已明確
- [ ] 目標用戶和使用情境已定義
- [ ] 功能範圍和優先級已確定
- [ ] 技術約束和依賴關係已識別
- [ ] 至少分析了 2-3 個設計方案
- [ ] 已選擇最佳方案並說明原因
- [ ] **已產出 proposal.md 文件**
- [ ] 用戶同意提案內容

## 下一步
